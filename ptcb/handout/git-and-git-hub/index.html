Git & GitHub

Git is a tool for version control. It helps you keep track of the changes that you and others do to your code. It is especially useful when several people collaborate on the same project. Git prevents people from accidentally overwriting each others code.

Git is very useful when keeping track of your own code even if you working independently. Imagine you accidentally break your code by deleting some lines. Instead of trying to figure out what you deleted you can go back in history with git and see previous versions of the code. You must add and commit changes to store these file changes on Git.

GitHub

If you haven’t done so already, go to GitHub and create an account there. From now on you will use this account daily, other coders will see this account and it represents your work as a coding individual. So choose your username wisely.

GitHub is a platform we can upload our code to via git and share it with others. Everybody can go through your public repositories and look at your code.

It is also a requirement for you to upload every exercise that you complete here at Xccelerate to GitHub, so that we can give you feedback.

Later on when you work on group projects, you’ll be using GitHub to collaborate.

Using Git

We assume that you have already installed Git on your machine. Check if it is working by typing this in the terminal:

1
2
git --version
git version 2.x.x

Set up Git

1
2
git config --global user.name "First Last"
git config --global user.email "my@mail.com"

It is important that your email is the same as the one you used for GitHub

Now we need to connect your GitHub with your git. For that we require your SSH key.

SSH key is used to securely establish a connection through encryption. GitHub can verify that you are who you claim to be through looking at your SSH key. Using the SSH protocol, you can connect and authenticate to remote servers and services. With SSH keys, you can connect to GitHub without supplying your username or password at each visit.

This key consist out of two parts, the public and private key. You can share your public key, but the private key is only meant for you.

Let’s check if you already have an SSH key generated, in your terminal type:

1
2
cd ~/.ssh
ls

If you can see in the listed document a file called id_rsa.pub you already have a SSH key generated and the please SKIP the next step. This is extremely important otherwise you overwrite your current SSH key. If you do not see the file it means you don’t have a key. This is how we can generate one.

1
ssh-keygen -t rsa -C my@mail.com

again use your GitHub email.

Copy the key and Go to GitHub settings > SSH and GPG keys > New SSH key and paste your public key that we just created.

Afterwards create a new repository on GitHub call it demo

Go back to your terminal, let’s change into the code folder we created recently in the terminal demo. In there we create a demo folder with a file called README.md inside, then initialize git.

echo "# demo" >> README.md is another way to create files, if you look at the README.md file in your editor you will notice it contains a # demo line. 
git init initializes a new repository on your local machine.

Let’s add our README.md file to git

1
2
git add README.md
git commit -m 'first commit'

git add [file] specifies which files you want to add to the repository. If you have several files it is easiest to add files by just typing git add . which adds all files in the folder to git. ‘Add’ will only start the tracking process, but not properly engrave it to git, for that we need one more step. 
git commit -m '[message]' will permanently make a record of the file version, which you later refer to by finding the commit id. The message should be short but descriptive, so that others can understand what happened. ‘added code’ is not very helpful, where as ‘fixed recursion error’ is a lot more helpful when you keep adding on top of your code.

However, everything we have done so far is local, so let’s connect it to GitHub. For that we need the SSH link GitHub provided you when setting up your repository. You should replace username with your own username.

1
2
git remote add origin git@github.com:username/demo.git
git push origin master

git remote add sets up the connection to GitHub, we usually call this connection origin 
git push [connection name] [branch] pushes your version to GitHub. Master refers to the branch name, something we will talk about at a later stage.

Make sure to insert your GitHub username in the git remote command.

Git Flow

Once your repository is set up those are the three commands you’ll need to use to upload a new version to GitHub. Remember them, you’ll need them all the time.

1
2
3
git add .
git commit -m 'description'
git push origin master

.gitignore

There are certain files which you don’t want to upload to GitHub. For example you might have a file in which you store passwords. This would be a bad idea if someone got a hold of them - especially if they are publicly available on GitHub. You can put the names of these files into a file called ‘.gitignore’. Create one in the root directory of you repository and add the relevant folders and files in there by name.

If you are using a Mac you might also want to include the ‘.DS_Store’ file. It’s a file Mac uses to record your folder preferences, there is nothing secretive in there but it is also pretty annoying to have on your GitHub.

Integration with Visual Studio Code

You can also use the third button on the left hand of the toolbar of Visual Studio Code which allows you to commit and push your files accordingly.

Branches

When you start working on a large project you don’t want to mess with the main code or source code, instead you want to make a copy of that code and start working on the copy instead. Once you are happy with your solution you can start to add your new code to the main code.

In order to achieve this separation of different codes we can use something called branches. Branches on GitHub are different copies of the same code with different changes applied to them. There will be times when you and several people are working on a project where everyone is going to be working on different branches. Once they have a feature or code that they are ready to commit, they will create a pull request. Other developers will check out the new code and if they agree with it, it will be merged into the main code.

Creating a new branch

1
git branch [branchname]

swapping into the new branch

1
git checkout [branchname]

Merging

Imagine you have completed writing your new code and it is ready to be introduced into the main codebase. So you want to get code from your working branch to the master branch. To do that we use a process called merging branches.

1
2
git checkout master
git merge [branch you were working on]

Et voila your new code has just been introduced into the master codebase.

This concept is done in our local machine. When working within a team of people or in a company you may need to set up pull requests mentioned above to verify code you are trying to add into your online source code.

Pull

Imagine you work on a project with other team members and someone has just merged a branch into the master branch. That master branch is now updated on GitHub, but you don’t have the new code on your local machine. To get that code you need to pull it from GitHub.

1
git pull origin [branch name]

Pull requests

If you work in a team you don’t just simply merge your branch into master, you want your team members to review your code. They need to agree that all new code is functioning before they allow you to merge the branch. The usual flow is to open a pull request on GitHub, which will inform other team members of your merge intentions.

Watch this video concerning Git Flow, and GitHub Pull Requests.

Merge conflicts

Sometimes when two developers have been working on the same section of a file, when they try to merge branches, Git doesn’t know which changes to merge. That results in a merge conflict, which is every developers nightmare. Git will mark the sections it cannot merge with a <<<<<<< and ======== as a divider between the two solutions. You have no choice but to manually go through your documents with a merge conflict and resolve the conflict by deleting one section and all of Git’s marks. Open the code in Visual Studios code to easily fix this merge issue. After you resolved the merge conflict add and commit again and you should be good to go.
